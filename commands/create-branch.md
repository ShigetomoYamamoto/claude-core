# ブランチ作成ルール

## 目的とプレフィックスの対応表

| 目的 | プレフィックス | 説明 | 例 |
|------|---------------|------|-----|
| 新規追加 | `feature` | 新しい機能や画面の追加 | feature/employee_export_csv_20251202 |
| 変更 | `update` | 既存機能の仕様変更・改善 | update/salary_calculation_logic_20251202 |
| 修正 | `fix` | バグ修正・不具合対応 | fix/attendance_time_display_20251202 |
| 削除 | `remove` | 機能や要素の削除・非表示 | remove/parent_company_alcohol_check_20251202 |
| リファクタリング | `refactor` | コード品質改善（機能変更なし） | refactor/dashboard_service_20251202 |
| 緊急修正 | `hotfix` | 本番環境の緊急対応 | hotfix/login_error_20251202 |

## AIによる目的判断ガイドライン（目的が「？」の場合）

詳細の内容から以下のキーワード・文脈で判断：

### `feature`（新規追加）と判断するケース
- 「〜を追加する」「〜を新設する」「〜機能を作る」
- 「新しい〜」「〜画面を作成」
- 現在存在しないものを作る

### `update`（変更）と判断するケース
- 「〜を変更する」「〜を改善する」「〜を修正する」（バグ以外）
- 「〜の表示を変える」「〜のロジックを変更」
- 「〜を〜に変える」
- 既存機能の仕様変更

### `fix`（修正）と判断するケース
- 「〜が動かない」「〜がおかしい」「〜のバグ」
- 「エラーが出る」「正しく表示されない」
- 「〜が〜になってしまう」（意図しない動作）
- 不具合・バグの修正

### `remove`（削除）と判断するケース
- 「〜を削除する」「〜を非表示にする」「〜を無くす」
- 「〜は不要」「〜を取り除く」
- 機能・要素・画面の削除

### `refactor`（リファクタリング）と判断するケース
- 「コードを整理」「リファクタリング」「可読性向上」
- 「パフォーマンス改善」（機能変更なし）
- 機能は変えずにコード品質を改善

### `hotfix`（緊急修正）と判断するケース
- 「緊急」「本番で〜」「至急対応」
- 本番環境で発生している問題

## ブランチ名の生成ルール

1. プレフィックスは上記対応表に従う
2. 目的部分はスネークケース（snake_case）で記述
3. 日付は YYYYMMDD 形式で自動付与
4. 形式: `{プレフィックス}/{目的の要約}_{日付}`

## 使用例

```
目的：新規追加
詳細：従業員一覧のCSVエクスポート機能を追加する
→ feature/employee_csv_export_20251202

目的：変更
詳細：給与計算のロジックを新しい要件に合わせて変更する
→ update/salary_calculation_logic_20251202

目的：修正
詳細：勤怠時間の表示が夜勤の場合に正しく表示されない
→ fix/night_shift_attendance_display_20251202

目的：削除
詳細：親会社情報画面のアルコールチェック欄は不要のため非表示にする
→ remove/parent_company_alcohol_check_20251202

目的：？
詳細：ダッシュボードの読み込みが遅いので高速化したい
→ update/dashboard_performance_improvement_20251202
（パフォーマンス改善で機能変更を伴う可能性があるためupdate）

目的：？
詳細：ログイン画面でエラーが出て入れない
→ fix/login_error_20251202
（バグ修正のためfix）
```

## 実行手順

1. 現在のブランチを確認
2. developブランチに切り替え
3. developブランチを最新化（git pull origin develop）
4. 新しいブランチを作成して切り替え
5. 結果を表示

⚠️ **注意**: ブランチ作成・切り替え後にはいかなる作業も行いません。必要な作業は別途手動で実行してください。
