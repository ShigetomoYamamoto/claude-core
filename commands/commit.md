# コミット手順

## ステップ 1: 変更内容の確認

以下を並行して実行して現状を把握する：

- `git status` — 変更・未追跡ファイルの一覧
- `git diff` — unstaged の差分
- `git diff --cached` — すでに staged の差分

## ステップ 2: コードレビュー

ステージング前に **code-reviewer エージェント** を起動して変更差分をレビューする。

- CRITICAL / HIGH の指摘があれば **コミットを中断し、先に修正する**
- MEDIUM 以下の指摘はユーザーに提示した上でコミットを続行してよい

さらに、差分に以下のいずれかが含まれる場合は **security-reviewer エージェント** も追加で起動する：
- 認証・セッション・トークン処理
- ユーザー入力のバリデーション・サニタイズ
- API エンドポイントの追加・変更
- シークレット・環境変数の参照
- 決済・課金・個人情報の処理

## ステップ 3: ステージングするファイルの選択

### 原則

- **すべての変更が同一の目的**（同じ fix/feat/chore）に属する場合 → `git add -A` でまとめてステージング
- **目的が混在**する場合（例: バグ修正とリファクタリングが混在）→ 関連ファイルだけを個別に `git add <file>` してコミットを分割する
- `.env` や認証情報ファイルは **絶対にステージングしない**。含まれていた場合は警告してスキップする

### 注意事項

- `git add .` や `git add -A` は意図しないファイルを含む恐れがあるため、変更ファイルが多い場合は内容を十分確認してから使用する
- バイナリ（画像・音声）の変更は、意図した変更かどうかを差分から判断して含める

## ステップ 4: コミットメッセージの生成

Conventional Commits のルール・type 一覧・生成例は `~/.claude/skills/git-workflow/SKILL.md` を参照すること。

### メッセージ生成ルール

1. 差分の内容とヒント（指定された場合）から type を判断する
2. description は **日本語** で「何をしたか」を端的に記述する（50文字以内目安）
3. 変更の背景や理由が非自明な場合は body に補足する
4. Co-Authored-By 行は **追加しない**（Attribution disabled）

## ステップ 5: コミット実行

```bash
git commit -m "$(cat <<'EOF'
<生成したメッセージ>
EOF
)"
```

コミット完了後、以下を表示する：

- コミットハッシュ（短縮形）
- コミットメッセージ
- 変更ファイル数・行数のサマリ

## ステップ 6: 結果確認

`git log --oneline -5` で直近5件のコミット履歴を表示し、問題なくコミットされたことを確認する。

## 注意事項

⚠️ **コミット後のプッシュは行いません。** プッシュが必要な場合は別途 `git push` を実行してください。

⚠️ **コミットメッセージを確定する前に必ず提示し、ユーザーの承認を得ること。**
