# ADR-014: Loop Engineering を「規律(原理)」として再定義する

**ステータス**: accepted

**日付**: 2026-06-20

## コンテキスト

「loop engineering」という語が、これまで3つの異なる対象を同時に指していた:

1. **原理**(検証付き閉ループで開発する考え方)
2. **1つのスキル**(`skills/loop-engineering/` = ミクロ実装)
3. **システム全体**(README が「Loop Engineering 運用」と呼ぶ5層)

この多義性が原因で、設計の議論が層を取り違えたまま進み、次の矛盾が温存されていた:

- **scope/サイズの判断が二重化**: `/autorun` の入口判断と `skills/loop-engineering` の STEP0(A/B/C)が、同じ「どれだけ大きい仕事か」を別々に判断していた。
- **入口の二重割り当て**: 素の自然言語(「直して」「作って」)が、README 上で `/autorun` と `loop-engineering` の両方の入口に割り当てられていた([ADR-007](./007-autonomous-loop-execution.md) / [ADR-008](./008-orchestration-declarative-flow.md) の実装に跨る)。
- **検証ゲートの定義揺れ**: `docs/requirements.md` は「検証ゲート=/verify-loop」と書く一方、`rules/autorun-flow.md` の verify フェーズは `/review-loop` を使う。

根本原因は、個々の部品の良し悪しではなく、**部品が奉仕すべき "一段上の定義" が明文化されていなかった**ことにある。本 ADR はその定義を確定する。

## 検討した選択肢

「loop engineering」という語を何に固定するか:

1. **規律・原理そのもの**(全スコープを貫く考え方)。スキル/コマンド/workflow は "その実装の一例"。
2. **特定の1スキル**(ミクロ実装)。システム全体は別名にする。

選択肢2は、マクロ(`/autorun`)・ミクロ(コードのループ)・レビュー(`/review-loop`)を別物として説明し続けることになり、層の取り違えが再発する。選択肢1なら、それらを「**同じ原理を別スコープに適用したもの**」と一言で説明でき、矛盾が構造的に起きにくい。

## 決定

**選択肢1(規律・原理そのもの)を採用する。**

### 定義(一文)

> **Loop Engineering = どんな変更も「先に決めた・機械で判定できる・ごまかせない完了条件」で囲い、合格するまで自律で回す規律。** 核は "閉じたループ(自己申告でない検証で閉じる)"。

### 不変条件(これを満たさなければ Loop Engineering でない)

スキルでもコマンドでもなく、**どの実装でも必ず真でなければならない4条件**:

1. **完了条件が、先・機械判定・改竄不可** — 後出しでも気分でもなく、テスト等で ○× が付き、途中で緩めない。
2. **作る者 ≠ 判定する者** — 自分の答案を自分で採点しない(最小規模なら "自分で書いた独立テスト" が判定者でよい)。
3. **有界** — 必ずハードストップ(回数/時間/予算)を持つ。ブレーキ無しは禁止。
4. **方向性と不可逆は人間** — 機械で決められない判断と、取り返しのつかない操作は人間に残す。

1〜2が「ループの正しさ」、3〜4が「ループの安全」。安全規律の正本は [`rules/loop-safety.md`](../../rules/loop-safety.md)。

### 統一モデル(再帰)

すべての段は同じループ「**決める(完了条件)→ 作る → 検証する → 直す →(合格 or ハードストップ)**」。段ごとに変わるのは2つだけ:

1. 「何を作るか」がどれだけ未確定か(未確定なら、それを決めるループが上に積まれる)
2. 検証する者が **機械か、人間か**

| 段(rung) | 完了条件 | 検証者 |
|---|---|---|
| 要件を決める | 作るべきものが合っている | 人間(機械判定不可ゲート) |
| 設計を決める | 設計が妥当 | 人間(判断不要ならスキップ) |
| 計画を立てる | 手順とファイルが揃う | ほぼ機械=自動 |
| コードを作る | 振る舞いが正しい(テスト緑) | 機械=自動 |
| レビュー | 指摘ゼロ | 独立した別者=自動 |
| 出荷(PR/デプロイ) | 世に出た | 人間(不可逆ゲート) |

横串(全段共通):安全 / メモリ(ラン間学習)/ 並列(worktree隔離)/ ドキュメント同期。

### 2つの原則(矛盾を構造的に消す)

- **入口は1つ・判断者は1人**: scope/サイズの判断は入口で1回・調査に基づいて行う。下流(例:`loop-engineering` STEP0)は、上流(`/autorun`)から scope を渡されたらそれを採用し、**再判定しない**。
- **ゲートは "置く" のでなく導出する**: 完了条件が機械判定可→自動、機械判定不可(=方向性)or 不可逆→人間ゲート。[ADR-007](./007-autonomous-loop-execution.md) の関門4点(要件/設計/PR/デプロイ)はこの規則の帰結であって、恣意的な選定ではない。

## 結果

### Positive

- マクロ/ミクロ/レビューを「同一原理の別スコープ適用」で一貫説明でき、層の取り違えが起きにくい。
- ゲートが導出規則から決まるため、関門の増減が機械的に判断できる。
- 既存資産はほぼ再利用可能(2026-06-20 の全資産監査で **廃止=0件**)。本 ADR は破棄でなく「定義の明文化 + 是正」を指示する。

### Negative

- 既存実装に「定義に対する逸脱」が点在しており、是正が要る(下記)。是正は8パターンに収束する。
- 不変条件2(作る者≠判定する者)は、これまで `loop-safety.md` に明文がなかった。本 ADR を機に Precondition へ追加する。

### 是正対象(2026-06-20 監査の集約。詳細は `docs/loop-engineering-audit-2026-06-20.md`)

| # | パターン | 主な対象 |
|---|---|---|
| P1 | 不変条件2が憲法に無い/一部で破れている | `loop-safety.md`・`agents/security-reviewer.md`(Write/Edit 保持) |
| P2 | 不変条件3(有界)がエージェント単体に無い | security-reviewer・tdd-guide・deploy/migration/rollback-runner |
| P3 | 「コードを作る」段の入口が3つ | loop-engineering / `/tdd` / tdd-guide |
| P4 | scope/サイズ判断の二重化 | STEP0 ↔ autorun入口、設計要否が autorun-flow ↔ plan.md |
| P5 | 薄ラッパー command が agent と重複 | requirements / analyze-task / design |
| P6 | 物理層(hooks)が不可逆と一致しない | mass-delete-blocker / secret-detection(非ブロック)、deploy/migrate/rollback(物理層なし) |
| P7 | verify-loop のドキュメント矛盾・未配線 | `requirements.md` ↔ `autorun-flow.md` |
| P8 | architect が要件段を再実装・人間ゲート内在化なし | `agents/architect.md` |

加えて穴(不足):要件段の受け入れ基準 → `loop-engineering` の VISION 述語への**受け渡し契約**が未定義。
