# ADR-001: グローバル / プロジェクトの2層構造を採用する

**ステータス**: accepted

**日付**: 2026-05-21

## コンテキスト

Claude Code を「どのプロジェクトでも知識を持って動く状態」にするには、設定の置き場所が必要。すべてをグローバル（`~/.claude/`）に置く案と、すべてをプロジェクト（`.claude/`）に置く案、両者を組み合わせる案がある。

## 検討した選択肢

1. **グローバル一本化** — どのプロジェクトでも同じ設定。利点は単純さ。欠点はプロジェクト固有のスタック規約を共通設定に混ぜざるをえず肥大化する
2. **プロジェクト一本化** — 各プロジェクトに `.claude/` を都度作る。利点は完全な独立。欠点は新規プロジェクトのたびに全部書き直しで時間がかかる
3. **2層構造** — マシン横断で不変の設定はグローバル、プロジェクト固有はプロジェクト側

## 決定

**2層構造を採用する。**

- グローバル（このリポジトリ）: スタック非依存の汎用設定（agents・commands・rules・hooks・skills・workflows・wiring）
- プロジェクト（`.claude/`）: スタック固有の設定（言語別 hook・スタック別ルール・デプロイ先）

プロジェクト側は `/init-autonomous` で生成する。

## 結果

### Positive

- 新規マシンでも `setup.sh` 1 発でグローバル設定が即座に揃う
- プロジェクト固有設定はそのプロジェクトに自然にコミットされ、チーム共有可能
- グローバル側の改善が `git pull && setup.sh` で全プロジェクトに即時反映される
- 役割分担が明確で、どこに何を置くか迷いにくい

### Negative

- 設定が2箇所に分散するため「どっちに書くか」の判断が必要
- グローバル側を更新したとき、各マシンで `setup.sh` を実行する必要がある
- プロジェクト側 `.claude/` がスタックごとに少し違う形になる
