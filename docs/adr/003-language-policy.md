# ADR-003: rules・agents は英語、commands は日本語

**ステータス**: accepted

**日付**: 2026-05-21

> **注記（2026-07-04）**: rules の言語方針（英語のみ）は [ADR-021](./021-language-policy-update.md) により「英語基調＋運用者向け日本語注記可」へ改定。agents=英語 / commands=日本語 は不変。

## コンテキスト

すべてのファイルを英語に揃える案と、ユーザーが日本語話者なので全て日本語にする案がある。Claude 自身は両言語で動作するが、ユーザー体験とモデル指示精度のバランスをどう取るか判断が必要。

## 検討した選択肢

1. **全英語** — 一貫性が高いが、ユーザー（日本語話者）にとってコマンド一覧が読みにくい
2. **全日本語** — ユーザーフレンドリーだが、モデルへの指示精度が下がる傾向。OSS 化の障害にもなる
3. **バイリンガル併記** — 全部を2言語で書く。保守コストが倍増し、片方が古くなる
4. **役割で分離** — モデル向け（rules・agents）は英語、ユーザー向け（commands）は日本語

## 決定

**役割で言語を分離する**:

- `rules/`・`agents/` → **英語**: モデルへの指示・国際的によく使われる用語（Conventional Commits・TDD・SOLID 等）を扱うため。Claude モデルが英語の指示で精度を出しやすい
- `commands/` → **日本語**: ユーザーが直接読んで起動するため。コミットメッセージ description も日本語

## 結果

### Positive

- モデルへの指示は英語で精度を維持できる
- ユーザーが日常使うコマンド説明は日本語で読みやすい
- 役割の境界が言語で視覚的にもわかる
- 英語の rules / agents は将来 OSS 公開しても他言語話者にそのまま使える

### Negative

- 言語が混在することへの違和感（プロジェクト全体で1言語が望ましいという意見もある）
- 一部の概念（`feat:` `fix:` 等の type 名）が rules（英語）と commands（日本語説明）の両方に登場し、表記揺れの懸念
- 翻訳ミスマッチが起きる可能性（rules の更新が commands に反映されない等）
