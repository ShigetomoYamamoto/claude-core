# ADR-013: 公式プラグインはレイヤーで取捨選択する

**ステータス**: accepted

**日付**: 2026-06-15（原則を起草） / 2026-06-20（ADR-012 の実装に整合し ADR-013 として確定）

> 本 ADR は「公式プラグインを採るか自作を残すか」を判断する**一般原則**。具体的な適用（git/レビュー/セキュリティの委譲先・維持対象）は [ADR-012](./012-official-plugins-for-git-review-security.md) を参照。

## コンテキスト

dotfiles として「公式からリリースされているものはなるべく取り込み、自作部分を少なくする」方針を検討した。自作資産は自分で保守し続ける必要があり、公式はアップデート・バグ修正が自動で乗るため、原則としては健全な方針である。

この検討は claude-code-setup プラグイン（コードベースを分析して自動化を提案するツール。自作の `/init-autonomous` が機能的に上位互換と判断した）の評価を発端とする。続いて、方法論に直接関わる複数のプラグイン（commit-commands・pr-review-toolkit・security-guidance・feature-dev・hookify）を自作の commands / agents / hooks と比較調査した。判断基準を持たないまま個別に「乗り換えるか / 自作を残すか」を決めると、プラグインが増えるたびに場当たりの判断を繰り返すことになる。

## 検討した選択肢

1. **公式優先・全面置換** — 公式を最大限採用し自作を捨てる。保守コストは最小。ただし公式は汎用・不特定多数向けに作られ、本リポジトリの方法論（コミット前承認・develop 固定・base=develop・TDD 強制 80%・日本語報告・破壊操作の動的ブロック）を設計上知らない。調査では commit-commands が「確認テキストの出力を禁止」する指示を持ち、承認ゲートと真っ向衝突した
2. **全面自作維持** — 公式を使わず全て自作。方法論は完全に守れるが、MCP サーバや LSP のような方法論非依存の機能まで自作するのは車輪の再発明で、保守負担が過大
3. **レイヤーで分離** — 「土台レイヤー」は公式優先、「方法論レイヤー」は方法論の*保証*を自作（hooks）で持ち、*振る舞い*は公式に寄せる

## 決定

**レイヤーで分離する。**

- **土台レイヤー**（スタック・方法論に依存しない汎用機能: MCP サーバ・LSP・UI 生成・セキュリティガイダンス等） → **公式を優先採用**し、自作しない
- **方法論レイヤー**（本リポジトリの開発思想を体現する部分: commit/PR/branch・コードレビュー・機能開発フロー・安全ガード） → 方法論の**【保証】**（コミット前承認・develop 固定・base=develop・秘匿除外・破壊操作ブロック・コミット規約）は**自作で持つ**が、汎用の**【振る舞い】**（コミットの実行・コードレビューそのもの）は公式に寄せてよい

**判断基準**: 公式プラグインは汎用・不特定多数向けに作られ、本リポジトリの方法論を構造的に知り得ない。ゆえに方法論の**保証**を公式に委ねてはならない。**ただし、その保証を hooks に決定的に集約すれば（[ADR-012](./012-official-plugins-for-git-review-security.md)：「危険操作は hooks／振る舞いは公式」）、方法論レイヤーの振る舞いを公式に寄せても保証は保たれる。** 土台レイヤーは方法論非依存のため、公式が保守性・品質で勝る。

これにより最終形は **「危険操作は hooks で決定的に制御／振る舞いは公式に寄せる／公式に良い同等品が無いものだけ自作維持」** となる（ADR-012 で実装）。

### 各プラグインの最終判断（ADR-012 で実装）

| プラグイン | レイヤー | 最終判断 |
|---|---|---|
| commit-commands | 方法論（振る舞いは汎用） | `/commit`→`/commit-commands:commit`、`/delete-merged-branches`→`/clean_gone` を採用。`/create-pr` は公式に良い同等品が無く（`commit-push-pr` は base=develop 強制 hook と衝突・無確認 push/PR）自作維持。承認・develop・秘匿・コミット規約は hooks が担保 |
| pr-review-toolkit | 方法論（レビューは汎用） | `code-reviewer` エージェントを採用（自作 `code-reviewer` を撤去）。`/review-loop` 系のエンジン `reviewer`/`fixer` は自作維持 |
| security-guidance | 土台 | 採用（編集時パターン警告＋commit 時 LLM レビュー）。呼び出せる agent が無いため、ループ用の `security-reviewer` agent と確定的な `secret-detection`/`git-add-secret-blocker` hook は自作維持 |
| feature-dev | 方法論 | 不採用（TDD 強制・カバレッジ 80%・2 モード・関門を構造的に満たさない）。`code-explorer` 等の観点は将来取り込み候補 |
| hookify | メタツール | 不採用（動的判定を要する既存フックは移植不可）。会話から軽量 regex ルールを量産する用途は追加候補 |

## 結果

### Positive

- プラグインが増えても「土台か／方法論の*保証*か*振る舞い*か」で機械的に判断でき、場当たりを避けられる
- 土台レイヤーの保守を公式へ委譲でき、自作の保守対象を「方法論の保証（hooks）＋オーケストレーションの核」に絞れる
- 品質ゲート・安全性（方法論の保証）が公式採用によって薄まるのを防げる（hooks が決定的に担保）

### Negative

- 「土台か方法論か」「保証か振る舞いか」の線引きが曖昧なケースでは個別判断が必要
- 方法論レイヤーの*振る舞い*を公式に寄せた分、各マシンで `/plugin` 導入が前提になる
- 方法論の保証（hooks）と、公式に同等品が無い部分（`/create-pr`・`reviewer`/`fixer`・`security-reviewer`・各 runner）の保守は引き続き自作で負う
