# ADR-007: 自走実行モデル（関門付きループ自走）を採用する

**ステータス**: accepted

**日付**: 2026-06-18

## コンテキスト

このリポジトリの目的は「AI エージェントが自走して開発を完成させる」こと（README 冒頭・`docs/requirements.md`）。

しかし既存実装は、全自動モード・サポートモードのどちらも、各フェーズコマンド（`/requirements`・`/design`・`/plan`・`/tdd`・`/commit` 等）が**フェーズ完了ごとに承認待ちで停止**し、人間が次のコマンドを打って初めて進む構造だった。フェーズを自動で連結する層が存在せず、「自走」は看板で実装が伴っていなかった。

後から追加した `rules/loop-safety.md`（安全装置）・`/autorun`・`/verify-loop` も、各コマンドの停止設計とオーケストレーション層の不在という根本を解いておらず、表面的だった。

「自走できる作り」にするには、次の3つが要る:
1. どこを自動で繋ぎ・どこで人間が確認するか（関門）の定義
2. フェーズを繋ぐオーケストレーション層
3. 既存コマンドの停止設計の見直し

本 ADR は **(1) を定義する**（(2)(3) は後続の設計・実装で扱う）。

## 検討した選択肢

関門（自走中に人間が確認する場所）をどこに置くか:

1. **方向性＋不可逆操作で確認** — 要件確定・設計確定（機械判定できず手戻りが最大）と PR 作成・デプロイ（不可逆）で人間確認。間の計画・実装・検証・コミットは自動連結
2. **不可逆操作のみで確認** — PR・デプロイ・外部送信でのみ確認。要件・設計も自動。関門最小・自動化最大だが要件ミスの手戻りリスク大
3. **完全自走** — デプロイ含め全自動。loop-safety のハードストップと破壊操作 hooks のみが歯止め。最も無人だが最もリスク大

## 決定

**選択肢1（方向性＋不可逆操作で確認）を採用する。**

- **全自動モードとサポートモードは別機能ではなく、1つの自走エンジンに与えるパラメータ（起点・ゴール）の違い**とする。

  | | 全自動モード | サポートモード |
  |---|---|---|
  | 起点 | 自由形式の目標 | 具体タスク / Issue |
  | 最初のフェーズ | 要件整理 | タスク分析 |
  | ゴール | デプロイ | PR |

- **関門（人間確認）は4点**: 要件確定・設計確定・PR 作成・デプロイ。
- **関門以外（計画・実装(TDD)・検証・コミット）は人間のコマンド打鍵なしで自動連結**する。
- 全区間で `rules/loop-safety.md`（ハードストップ・ゴールドリフト検知・不可逆操作確認）が作動する。
- 既存のフェーズコマンドは**自走の実行部品として再利用**し、停止設計は関門に該当するものだけ残す。

## 結果

### Positive

- プロジェクトの目的（自走）と実装が一致する
- 機械判定できない方向性（要件・設計）と不可逆操作だけ人間が見るので、手戻りと事故を抑えつつ自動化できる
- 2モードを1エンジンに統合でき、保守箇所が減る
- `rules/loop-safety.md` がそのまま全モードの安全装置として機能する

### Negative

- オーケストレーション層の新規実装と、既存コマンドの停止設計の見直しが必要（実装コスト）
- 関門の間が自動で進むため、要件・設計の関門でのレビュー品質が全体品質を左右する
- 「単発でコマンドを打つ従来の使い方」と「自走」の2系統を併存させる設計判断が要る
